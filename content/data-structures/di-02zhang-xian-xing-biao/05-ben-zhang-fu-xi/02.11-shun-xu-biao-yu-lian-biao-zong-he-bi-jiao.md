## 条目概览

选择存储结构时不能只背“数组查找快、链表插入快”。必须把操作前提说完整：按位置访问、是否已经持有前驱、是否需要先查找、数据规模是否变化，以及内存和缓存成本。

> [!summary]
> 选择存储结构时不能只背“数组查找快、链表插入快”。必须把操作前提说完整：按位置访问、是否已经持有前驱、是否需要先查找、数据规模是否变化，以及内存和缓存成本。

## 1. 核心对比

| 比较项 | 顺序表 | 单链表 |
|---|---|---|
| 存储空间 | 连续 | 可分散 |
| 按下标访问 | $O(1)$ | $O(n)$ |
| 无序按值查找 | $O(n)$ | $O(n)$ |
| 表头插入 | $O(n)$ | 带头结点时 $O(1)$ |
| 表尾插入 | 有容量时 $O(1)$ | 有尾指针时 $O(1)$，否则 $O(n)$ |
| 已知前驱插入 | 仍可能移动元素 | $O(1)$ |
| 按位置插入 | $O(n)$ | $O(n)$ |
| 删除表头 | $O(n)$ | $O(1)$ |
| 已知前驱删除后继 | 仍可能移动元素 | $O(1)$ |
| 额外指针空间 | 无逐元素指针 | 每结点一个 `next` |
| 缓存局部性 | 通常较好 | 通常较弱 |
| 容量变化 | 静态受限，动态需扩容 | 按需申请 |

## 2. 为什么“链表插入快”有前提

若已经持有前驱 `p`：

```c
newNode->next = p->next;
p->next = newNode;
```

修改链接为：

$$
O(1)
$$

但若题目要求“在第 $i$ 个位置插入”，必须先从头寻找前驱：

$$
O(n)+O(1)=O(n)
$$

因此准确说法是：

> 链表在已知插入位置附近结点指针时，局部插入删除代价低。

## 3. 为什么顺序表遍历通常更快

虽然二者遍历的渐近复杂度都是：

$$
O(n)
$$

顺序表元素连续，CPU 更容易利用缓存；链表结点可能分散，每次还要读取指针。因此在真实机器上，顺序表的顺序遍历往往具有更小常数。

复杂度相同不代表实际性能完全相同。

## 4. 空间利用比较

顺序表的潜在浪费：

```text
预留容量 - 实际长度
```

链表的额外成本：

```text
每个结点的指针域 + 动态分配管理开销
```

哪一种更省空间取决于元素大小、预留容量和指针大小，不能一概而论。

## 5. 选择建议

### 优先顺序表

- 频繁按下标访问；
- 遍历和缓存性能重要；
- 数据规模相对稳定；
- 元素较小，不希望每个元素额外带指针；
- 中间插入删除较少。

### 优先链表

- 数据规模变化频繁；
- 经常在已知结点附近插入删除；
- 不需要随机访问；
- 难以一次申请大块连续空间；
- 需要通过重连结点高效拆分或合并。

### 优先双向链表

- 需要频繁向前和向后移动；
- 已知目标结点时经常删除；
- 可以接受更高空间和维护成本。

### 优先循环链表

- 处理周期性过程；
- 需要从任意位置继续遍历；
- 实现循环队列式链接或约瑟夫环模拟。

## 6. 典型场景判断

| 场景 | 推荐结构 | 原因 |
|---|---|---|
| 按编号频繁读取成绩 | 顺序表 | 随机访问 |
| 浏览器前进与后退记录 | 双向链表 | 双向移动 |
| 轮流调度任务 | 循环链表 | 周期遍历 |
| 已知任务结点后频繁插入 | 单链表 | 局部修改链接 |
| 大量排序和连续扫描 | 顺序表 | 缓存局部性好 |

## 7. 复杂度判断中的常见陷阱

1. 把按下标访问和按值查找混为一谈；
2. 只计算修改指针，不计算寻找结点；
3. 认为有尾指针后链表所有操作都是 $O(1)$；
4. 认为动态数组每次尾插都一定是 $O(1)$，忽略偶发扩容；
5. 只看大 $O$，忽略空间与缓存成本；
6. 认为双向链表按下标访问是 $O(1)$。
