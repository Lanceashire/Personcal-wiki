## 条目概览

IPython 是增强型 Python 交互式环境，Jupyter Notebook / JupyterLab 则把代码、输出、图表和文字说明组织在一个可保存的文档中。教材强调这些工具适合探索式数据分析：一边运行代码，一边观察中间结果，一边记录解释。

## 1. 概念概览

IPython 是增强型 Python 交互式环境，Jupyter Notebook / JupyterLab 则把代码、输出、图表和文字说明组织在一个可保存的文档中。教材强调这些工具适合探索式数据分析：一边运行代码，一边观察中间结果，一边记录解释。

## 2. 学习重点

- IPython 适合快速试验表达式、对象和函数。
- Jupyter 适合数据分析过程记录，但不等于工程代码组织的全部。
- 自省、补全、历史命令和魔法命令能提高探索效率。
- 对期末复习而言，交互式环境最适合验证运行结果判断题。

## 3. 规范定义

| 工具 | 作用 | 典型场景 |
| --- | --- | --- |
| IPython shell | 增强交互式命令行 | 快速试验语法、对象、函数 |
| Jupyter Notebook | 代码、输出、Markdown 混合文档 | 数据探索、课堂笔记、实验记录 |
| JupyterLab | 更完整的 Notebook 工作区 | 多文件、多 Notebook 项目 |
| Tab completion | 自动补全变量、属性、路径 | 探索对象方法和减少拼写错误 |
| Introspection | 查看对象信息 | `?`、`??`、`help()` |

## 4. 直觉解释

Notebook 像实验记录本：每个单元格是一小步实验，输出就是实验现象，Markdown 是解释。它适合学习和探索，但如果逻辑越来越复杂，就应该逐步整理成函数和脚本。

## 5. 原理与机制

Jupyter Notebook 的代码单元并不是从上到下自动保证状态干净。变量会保存在当前 kernel 中，反复运行不同单元格可能导致状态和页面顺序不一致。

示例：

```python
a = [1, 2, 3]
```

如果后面某个单元格运行：

```python
a.append(4)
```

再回头运行只读取 `a` 的单元格，就会看到已经被修改后的对象。这和对话中“变量名绑定同一个可变对象”是同一类问题。

## 6. 关键步骤、分类或公式

Notebook 使用建议：

1. 每个单元格只做一个清晰步骤。
2. 关键变量生成后立刻检查形状、类型或前几行。
3. 修改可变对象前确认是否需要复制。
4. 重要结果能从头到尾重启运行复现。
5. 最终项目把核心逻辑整理到 `.py` 脚本或模块。

## 7. 例题与具体案例

### 例题：Notebook 状态陷阱

单元格 1：

```python
data = [1, 2, 3]
```

单元格 2：

```python
data.append(4)
```

单元格 3：

```python
print(data)
```

如果单元格 2 被运行两次，单元格 3 输出：

```python
[1, 2, 3, 4, 4]
```

这不是 Notebook 错了，而是可变列表被原地修改了两次。

## 8. 易错点与边界情况

> [!warning] 易错点
> Notebook 单元格的视觉顺序不一定等于真实运行顺序。复习代码题时，如果结果奇怪，应重启 kernel 并从头运行。

常见问题：

- 变量来自之前运行过但现在看不到的单元格；
- 同一个可变对象被多次修改；
- Notebook 用的 kernel 和终端环境不同；
- 一段探索代码堆得太长，导致难以复现。
